---- Spacetime and The Scheduler ---- The FST format An FST source file contains a list of States. A State contains a list of Callbacks. A Callback contains these Blocks, in any order : -> A "do" Block; -> Optionally, a "to" Block; -> Optionally, a "when" Block; ---- Preprocessor #insert "file" #replace text some more text #by replaced text here #end Useful definitions : #replace yes #by 1 #end #replace no #by 0 #end #replace mul({{a}},{{b}}) #by ({{a}}*{{b}}) #end mul(2,3) => (2*3) ---- States ---- Callbacks This lists the available Callbacks. Copy Parameters : Depth Overwrite Delete ---- Path resolution In FST, all nodes (i.e the Lnode, as well as all of the Vnodes, Enodes and their variables) are located in the Global Tree, in a parent-children relationship. One can think of the Global Tree as a true filesystem, except that each directory is a file at the same time. The Tree structure has shown its efficiency for years : -> You use it everyday to manage your data and access resources; -> UNIX's "Everything is a file" philosophy makes programming on this platform quite easier. You need and interface to the System's Kernel, and you know there is a file for it you can read and write to (note that that file can be virtual); -> In programming languages, functions can be thought of as nodes that carry their own value and can call other nodes (The "main" function being the root node); -> That's what Object-Oriented Programming is about, after all; -> You can locate nodes by their absolute position in the hierarchy (eg. A global counter) or their relative position to the current node (eg. The chest near to your hero). -> You can rearrange nodes as they move into the world. -> This allows creating a game world like this : Universe `- Galaxy_0 `- SubGalaxy_0 `- SubGalaxy_1 `- Solar_System `- Venus `- Earth `- Continents `- Seas `- Lost_Island `- Hero `- Jupiter `- SubGalaxy_N `- Galaxy_1 `- Galaxy_N -> Need to share files ? Use a Network FileSystem. Need online multiplayer ? Do this with nodes. -> All the filesystem goodies come in, such as Symbolic Links (weak references) and Hard Links (strong references with reference counting) The following explains the semantics of a Path in FST's Global Tree. Node names Node names are strings of only alphanumerical characters from the UTF-8 character set. Underscores ('_') are allowed. No name may have less than one alphabetic character. The user is advised to stick to the following convention : -> User-created node names should start with an uppercase letter, as each word following it should do. -> Names consisting of no uppercase letter should be left for the Scheduler. No node name may be one of these reserved names : statelabel random x y z time rotation scale vel accel targets <number value> on off yes no true false Which are ordered like so : MyEnode `- x <= MyEnode's x axis's offset (relative to its parent's). `- vel <= MyEnode's x axis's offset's velocity. `- accel <= MyEnode's x axis's offset's velocity's velocity. `- rotation <= MyEnode's rotation around its x axis. `- vel <= MyEnode's rotation's velocity around its x axis. `- accel <= MyEnode's rotation's velocity's velocity around its x axis. `- scale <= MyEnode's x axis's scale. `- vel <= MyEnode's x axis's scale's velocity. `- accel <= MyEnode's x axis's scale's velocity's velocity. `- y <= MyEnode's y axis's offset (relative to its parent's). `- vel <= MyEnode's y axis's offset's velocity. `- accel <= MyEnode's y axis's offset's velocity's velocity. `- rotation <= MyEnode's rotation around its y axis. `- vel <= MyEnode's rotation's velocity around its y axis. `- accel <= MyEnode's rotation's velocity's velocity around its y axis. `- scale <= MyEnode's y axis's scale. `- vel <= MyEnode's y axis's scale's velocity. `- accel <= MyEnode's y axis's scale's velocity's velocity. `- z <= MyEnode's z axis's offset (relative to its parent's). `- vel <= MyEnode's z axis's offset's velocity. `- accel <= MyEnode's z axis's offset's velocity's velocity. `- rotation <= MyEnode's rotation around its z axis. `- vel <= MyEnode's rotation's velocity around its z axis. `- accel <= MyEnode's rotation's velocity's velocity around its z axis. `- scale <= MyEnode's z axis's scale. `- vel <= MyEnode's z axis's scale's velocity. `- accel <= MyEnode's z axis's scale's velocity's velocity. `- time <= MyEnode's time axis's offset (relative to its parent's). `- vel <= MyEnode's time axis's offset's velocity. `- accel <= MyEnode's time axis's offset's velocity's velocity. `- targets <= Filled by Hard Links to Enodes reached by one or more of MyEnode's Red HitBoxes in current time position. These Hard Links are removed at the beginning of the next step in time. MyEnode can copy one or more of them to a personal node to keep track of them. `- dnode <= Hard Link to MyEnode's Dnode. Separator and operators : -> The '.' symbol is the "child-of" operator. -> The '^' symbol resolves to the parent node of the previously resolved node in the path. Prefixes : -> The absence of prefix means "the current node". -> Paths starting with '%' are relative to the Lnode (which is also the root node). -> Paths starting with ':' are relative to the Root Vnode (it actually resolves to "%.root_vnode"). To remember it, think of ':' as a pair of eyes ("The view"). The Root Vnode is shared across the network. -> Paths starting with '&' are relative to the Root Enode (it actually resolves to "%.root_enode"). To remember it, think of '&' as a weird capital 'E'. The Root Enode is shared across the network. -> Paths starting with '@' are relative to the Root Dnode (it actually resolves to "%.root_dnode"). It is also referred to as the "Aether", where the Entities are loaded but not yet instanciated (aka "brought to existence"). -> Paths starting with '~' are relative to the executable's full path in the user's filesystem. (it actually resolves to "%.app_path"). The '~' character is used as a shortcut to the home directory on Linux. Wildcards : -> The '?' character is replaced at resolution-time by any alphanumerical character. -> The '*' character is replaced at resolution-time by one or more alphanumerical characters. See examples below : -> "foo" resolves to the node named "foo" that is a child of the current node, as do ".foo", "..foo", "...foo" and "......foo"; -> "&.Earth.Continents" resolves to the node named "Continents" that is a child of the node named "Earth" that is a child of the Root Enode, as do "&...Earth..Continents" and "&.Earth.^.Earth.Continents". -> "^.bar" resolves to the node named "bar" that is a child of the parent node of the current node, as do "...^..bar". -> "foo*bar" resolves to each node that is a child of the current node, and which name starts with "foo" and ends with "bar", as do "foo**bar" and "foo***bar". -> "foo?ar" resolves to each node that is a child of the current node, and which name starts with "foo", followed by a single character and "ar". Annex A : FST Preprocessor directives reference

File content insertion : #include "filename" directive

The #include "filename" directive shall be replaced by the content of the file whose path in the User's Filesystem is filename. The next end-of-line character following such a directive shall be preserved.
filename's characters shall be read "as-is", which means, for instance, that spaces and backslashes are read as they are and lose any other meaning.
filename shall not contain quote characters.
filename shall be given "as-is" to the User's Operating System, which will handle path resolution itself. It means, for instance, that slashes should be placed between a folder's path and a given file or sub-folder of that folder.

The presence of any character other than space between the #include token and the "filename" token shall cause an error.
The presence of any character other than space between the "filename" token and the end-of-line shall cause an error.
A filename pointing to a file that can't be accessed shall cause an error.
Implementations shall not have the number of levels of inclusion be under a fixed limit.
However, infinite inclusions shall be detected by implementations and cause an error. (eg. "A.fst" includes "B.fst", and "B.fst" includes "A.fst").

The implementation could detect infinite inclusions by building a directed graph and detecting cycles in it.

Content substitution : #replace, #by

The following FST code : 

#replace
before
#by
after
#end
shall cause the implementation to treat every occurence of before at if it was after, for the remaining of the preprocessing step.
before shall start right after the first end-of-line character following the #replace directive, and end right before the last end-of-line character preceding the #by directive.
after shall start right after the first end-of-line character following the #by directive, and end right before the last end-of-line character preceding the #end directive.

User-defined messages : #info, #warn, #error

The following FST code : 

#info
message
#end
shall be a hint to the implementation to present message to the user.
How and when message is presented is up to the implementation.
message could be printed in a special color at the very moment the #end directive is parsed.

The #warn and #error directives shall behave similarly, except that #error shall cause an error as an additional effect.
message shall start right after the first end-of-line character following the #info directive, and end right before the last end-of-line character preceding the #end directive.

Annex B : FST State Controllers reference

ChangeState

Causes the target to switch to a given state, and delays its execution until the next tick. The old state's execution is interrupted.

Required parameters:

value (State node)
The new state the target will switch to.

ExecState

Causes the target to execute another state.
To be accurate, it does three things :
1) Interrupts the target's current state execution;
2) Changes the target's current state;
3) Executes the new state immediately.
A too frequent use of ExecState is discouraged in favor of ChangeState.
It is only recommended to use it to execute small states that nearly act as functions.
